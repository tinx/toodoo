# ADR 0004: Authenticator Owns Its Own Datastore

**Status:** Accepted
**Date:** 2026-06-10
**Revised:** 2026-06-12 (security review, pre-implementation): session
and invite tokens are stored as SHA-256 hashes, not plaintext; CLI/MCP
tokens expire by default. See "Tokens are hashed at rest."
**Related:** ADR 0002 (privilege separation via systemd hardening),
ADR 0005 (request flow and TLS termination boundary)

## Context

toodoo runs as a set of privilege-separated services (Listener, TLS
Signer, Authenticator, Engine) for the reasons articulated in
`README.md` and `docs/SECURITY.md`. Each service runs in its own
hardened systemd sandbox with its own dynamic UID and its own
filesystem footprint (per ADR 0002).

An earlier draft of the data model placed all persistent state in a
single SQLite database owned by the Engine, including:

- `users.password_hash` — argon2id-encoded credentials
- `sessions` — bearer tokens / cookie values, expiry, client metadata
- `invites` — single-use signup tokens generated by admins

This co-location had a structural problem: the Authenticator is
supposed to be the *only* service that handles credential material,
but credentials were stored where the Engine could read them. To
actually authenticate users, the Authenticator then needed one of two
things, both of which defeat its isolation:

1. **Direct database access** to the Engine's SQLite file. The
   Authenticator and Engine would share a filesystem path and a
   lockfile, putting credentials inside the Engine's blast radius
   and defeating the systemd-level filesystem separation between them.
2. **RPC every credential operation through the Engine.** The
   Authenticator asks the Engine "is this password right?" and "store
   this session," which makes the Engine a credential-handling
   service. Same problem with a longer code path and an additional
   trust transition.

Neither option leaves the Authenticator a meaningful security
boundary. The privilege separation is theatre if the data it's
supposed to confine isn't actually confined.

## Decision

**The Authenticator owns its own SQLite database** in its own
`StateDirectory=` (e.g. `/var/lib/toodoo-authenticator/auth.sqlite`
under the Authenticator's dynamic UID). The Engine owns its database
in its own `StateDirectory=` (e.g. `/var/lib/toodoo-engine/data.sqlite`
under the Engine's dynamic UID). **The two databases are separate
files. They never share a directory, never share a lockfile, and
neither service ever opens the other's file.** They are linked only
by `user_id` as a logical reference.

### Engine's database (per `docs/specs/data-model.md`)

Holds business identity and business data:

```
users          (id, handle, display_name, created_at, disabled_at)
lists          (id, owner_user_id, title, created_at, archived_at)
list_members   (list_id, user_id, role, added_at, added_by)
items          (id, list_id, text, done, due_at, priority, position,
                version, created_at, updated_at, deleted_at, created_by)
labels         (id, list_id, text, color, description, created_at, created_by)
item_labels    (item_id, label_id)
schema_version (version, applied_at)
```

No credential material. No sessions. No invites.

### Authenticator's database (per `docs/specs/authentication.md`)

Holds authentication state:

```
identities
  id          TEXT PRIMARY KEY              -- internal
  user_id     TEXT NOT NULL                 -- logical FK to Engine's users.id
  kind        TEXT NOT NULL                 -- 'local' | 'oidc'
  created_at  INTEGER NOT NULL

local_credentials
  identity_id   TEXT PK REFERENCES identities(id) ON DELETE CASCADE
  password_hash TEXT NOT NULL                -- argon2id encoded string
  must_change   INTEGER NOT NULL DEFAULT 0   -- set by admin password reset
  updated_at    INTEGER NOT NULL

oidc_links                                    -- schema present in v1; v2 enables OIDC
  identity_id  TEXT PK REFERENCES identities(id) ON DELETE CASCADE
  issuer       TEXT NOT NULL                  -- e.g. "https://accounts.google.com"
  subject      TEXT NOT NULL                  -- opaque per-issuer ID
  email_seen   TEXT                           -- last-seen, not authoritative
  linked_at    INTEGER NOT NULL
  UNIQUE (issuer, subject)

sessions
  token_hash   TEXT PRIMARY KEY               -- SHA-256 of the bearer /
                                              -- cookie value; plaintext
                                              -- token is never stored
  user_id      TEXT NOT NULL
  identity_id  TEXT NOT NULL REFERENCES identities(id)
  client_kind  TEXT NOT NULL                  -- 'web' | 'cli' | 'mcp'
  user_agent   TEXT
  created_at   INTEGER NOT NULL
  last_seen    INTEGER NOT NULL
  expires_at   INTEGER NOT NULL

invites
  token_hash  TEXT PRIMARY KEY                -- SHA-256 of the invite token
  created_by  TEXT NOT NULL                   -- user_id
  created_at  INTEGER NOT NULL
  expires_at  INTEGER NOT NULL
  redeemed_at INTEGER
  redeemed_by TEXT
  note        TEXT

schema_version (version, applied_at)
```

### Tokens are hashed at rest

Session and invite tokens are stored as **SHA-256 hashes**, never in
plaintext. The plaintext token exists only in the client's possession
(and transiently in the Authenticator's memory during issuance and
validation). Lookup is hash-then-index: the Authenticator hashes the
presented token and queries by `token_hash` — one cheap hash per
request, no measurable cost.

Rationale: this database is included in backups (see
`docs/specs/backup-restore.md`), and backups travel — off-host
storage, object stores, removable media. A stolen backup containing
plaintext session tokens is an instant account takeover of every
active session, strictly worse than the argon2id password hashes
stored next to them. Hashing also protects against read-only
compromise of the Authenticator's filesystem or a leaked database
file. (SHA-256 without salt/stretching is sufficient here: tokens
are 32 bytes of CSPRNG output, not low-entropy passwords —
brute-forcing the preimage is infeasible.)

Token lifetimes: web session cookies default to 30 days rolling.
**CLI/MCP bearer tokens default to 90-day expiry** with
CLI-assisted rotation (`toodoo auth rotate`); non-expiring tokens
are an explicit operator opt-in, not the default.

### Why an `identities` table (not columns on a user record)

The canonical stable OIDC identifier is **`(issuer, subject)`**, not
email. `sub` is opaque per-issuer (Google numeric, GitHub numeric,
Azure UUID); emails change, can be unverified, may be absent.

Modeling identities as their own table buys three things:

1. **One user, multiple identities**: a single toodoo `user_id` can
   have a local password identity *and* one or more OIDC identities,
   all resolving to the same business user. No duplicate user records
   when someone links a second login method.
2. **OIDC drops in with zero schema churn**: ship `oidc_links` empty
   in v1; v2 turns on the OIDC adapter and starts populating it. No
   migration required at the v1→v2 boundary.
3. **Identity vs. credential separation is explicit**: a `local`
   identity has a `local_credentials` row; an `oidc` identity has an
   `oidc_links` row. Schema enforces "exactly one credential shape
   per identity," and switching shapes requires explicit work.

### Cross-service coordination

Three operations cross the database boundary. All are infrequent and
operator-/user-initiated, never per-request:

- **Signup (invite redemption)**:
  1. User presents invite token + chosen handle + display_name + password.
  2. Authenticator validates the invite token in its own DB.
  3. Authenticator asks Engine via an internal RPC (`POST
     /internal/users` over the existing Authenticator→Engine socket;
     see `docs/specs/authentication.md`, Cross-Service RPCs):
     "create user with handle=X, display_name=Y" → Engine returns
     `user_id`.
  4. Authenticator creates an `identities` row (`kind='local'`,
     `user_id`), a `local_credentials` row, marks the invite
     redeemed, and issues a session.
  5. If step 3 succeeds and step 4 fails, the operator has a "ghost
     user" (Engine row exists, no identity). The signup endpoint
     retries; if the inconsistency persists, an admin command
     reconciles. Safe failure mode.

- **Login**:
  1. Authenticator looks up the local credential by handle (via
     Engine RPC to resolve handle → user_id, since handles live in
     Engine's DB), verifies the password.
  2. Authenticator asks Engine "is `user_id` enabled?" — login fails
     if Engine reports `disabled_at` is set. Cheap round-trip on the
     rare login path; avoids replicating disable state.
  3. Authenticator inserts a `sessions` row, returns the token.

- **Disable user**:
  1. Admin runs `toodoo admin user disable <handle>`.
  2. Engine sets `disabled_at` on the user row.
  3. Authenticator deletes all `sessions` rows for that `user_id`.
  4. Partial-failure safety:
     - If only step 2 succeeds: user has stale sessions, but Engine
       rejects every request as `403 disabled` (defense-in-depth
       check on `X-Toodoo-User-Id` header, see ADR 0005).
     - If only step 3 succeeds: sessions killed, but user can log
       back in — until login's "is user enabled?" check (step 2 of
       login) catches it.
     - Either way, the user is effectively disabled. The operator
       gets a clear error and can retry to make the state fully
       consistent.

## Consequences

### Positive

- **Real privilege separation at the data layer.** A compromise of
  the Engine reads all business data but no credentials, no live
  sessions, no OIDC links. A compromise of the Authenticator reads
  credentials and sessions but no lists, items, or shared content.
- **systemd filesystem isolation actually enforces the boundary.**
  Each service's `StateDirectory=` is owned by a distinct dynamic UID
  and is inaccessible to other services by ordinary filesystem
  permissions. The architectural decision has a mechanical
  enforcement layer.
- **OIDC is a v2 drop-in.** The schema is shaped for it from day
  one; no breaking migrations needed when OIDC ships.
- **Multiple identities per user** is a first-class capability, not
  a workaround. A future "link your Google account" feature is an
  insert, not a re-architecture.

### Negative (accepted)

- **Cross-service coordination on a small set of operations**
  (signup, login, disable). Each has a documented two-write pattern
  with safe partial-failure semantics. The complexity is real but
  contained, and the operations are rare.
- **Two databases to back up**, not one. The `toodoo-backup` CLI
  must snapshot both via their respective backup sockets
  (`backup.sock`, group `toodoo-backup` — see ADR 0006) and include
  both in the backup tar. See `docs/specs/backup-restore.md` (to be
  written).
- **Two schema versions to migrate independently.** Each service
  manages its own migrations and refuses to start against a too-new
  on-disk schema. Backup manifests record both schema versions.

### Not affected

- The Engine's API surface, ACL model, sync protocol — all unchanged.
  The Engine receives requests with a pre-validated `X-Toodoo-User-Id`
  header (see ADR 0005) and queries its own DB as before.
- The hexagonal `Storage` port abstraction. Both services have their
  own port and adapter; both can swap SQLite for Postgres
  independently if a future deployment wants that.
